# Demo instructions: running the secure CI/CD pipeline

This guide walks through an end‑to‑end demonstration of the secure CI/CD pipeline.  It assumes you have forked the repository into your GitHub account and configured repository secrets as described below.

## Prerequisites

1. **GitHub repository** – The secure pipeline workflow file (`.github/workflows/secure-ci-cd-pipeline.yml`) must be present in your repository’s default branch.  You already added this file in a previous step.
2. **Repository secrets** – Generate a Cosign key pair and add the private key and password as GitHub secrets:
   * `COSIGN_PRIVATE_KEY` – contents of `cosign.key` generated by `cosign generate-key-pair`.
   * `COSIGN_PASSWORD` – password used to protect the private key.
   Optionally add any `SEMGREP_APP_TOKEN` or `GITLEAKS_LICENSE` if you use paid services.
3. **Signed image repository** – The workflow uses GitHub Container Registry (ghcr.io) by default.  Ensure that packages permissions allow package publishing.
4. **Kubernetes cluster** – For deployment testing, have access to a Kubernetes cluster with `kubectl` configured.

## Running the pipeline

1. **Make a code change** – Edit a file in the repository (for example, update the FastAPI version or change a message in `app/main.py`).  Commit and push the change to the `main` branch or open a pull request.

2. **Trigger the workflow** – On push or pull request, GitHub Actions will run the `Secure CI/CD Pipeline` workflow.  The workflow performs:

   * Static analysis via Semgrep; it fails if critical vulnerabilities are found【71470937592858†L71-L87】.
   * Software composition and IaC scanning with Trivy, failing on high or critical vulnerabilities.
   * Secret scanning with Gitleaks to detect hard‑coded secrets.
   * Policy enforcement with Conftest against `config/deployment.yaml`.  The included policy `policy/latest-tag.rego` denies deployments that use the `:latest` tag on container images【71470937592858†L71-L87】.
   * Build, push and sign the Docker image.  The image is tagged with the branch name and pushed to `ghcr.io/<owner>/<repo>:<branch>`.  Cosign signs the image using the provided private key and uploads the signature to the registry【453300894240225†L446-L456】.

3. **Review results** – Navigate to the **Actions** tab in GitHub.  Open the workflow run to see logs for each job.  If any security gate fails, the build stops and deployment doesn’t occur.  SARIF files are uploaded to the Security tab so you can review detailed findings.

4. **Deploy** – After the image is built and signed, the final job prints a deployment placeholder.  To actually deploy, update `deploy` job steps in the workflow to run `kubectl` or `helm` commands.  Alternatively, you can deploy manually using the sample manifest:

   ```bash
   # Replace REPO with your GitHub repository name
   # Replace the image tag with the one printed in the job summary
   sed -i 's#ghcr.io/REPO/wiki-service-k8s:main#ghcr.io/your-user/wiki-service-k8s:main#' config/deployment.yaml
   kubectl apply -f config/deployment.yaml
   ```

5. **Verify deployment** – Once the pods are running, port‑forward the service and call the endpoints:

   ```bash
   kubectl port-forward deployment/wiki-service 8080:8000
   curl http://localhost:8080/
   # => {"status":"ok"}
   curl http://localhost:8080/build-info
   # => shows build metadata injected by the pipeline
   ```

## Live demo tips

To present a live demo:

1. **Start with a brief overview** of the pipeline and security gates using the slides provided (see `slides` directory or create your own).  Highlight the tools (Semgrep, Trivy, Gitleaks, Conftest, Cosign) and explain how each contributes to supply‑chain security.
2. **Show the repository on GitHub** and point out the workflow file, application code, Dockerfile, policy and Kubernetes manifest.
3. **Trigger a workflow run** by making a small change.  Explain how the pipeline progresses through the security checks, builds the image and signs it.  Demonstrate that the build stops when a vulnerability threshold is exceeded.
4. **Deploy the application** using the signed image and verify the `/build-info` endpoint returns the expected metadata.
5. **Conclude** by summarizing how policy enforcement and artifact signing improve trust in the release process.

By following these instructions, you will have a reproducible and auditable demonstration of a secure CI/CD pipeline in action.
